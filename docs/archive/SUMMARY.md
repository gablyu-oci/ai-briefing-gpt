## Top 5 Architectural Decisions

### 1. Ingest once / score once / deduplicate once / render differently per audience
The system builds one canonical daily bundle of 20-40 ranked story objects (shared computation), then each audience gets a Stage 2 pass that re-orders sections, adjusts commentary tone, and applies per-person word budgets. This means LLM generation costs scale with story count, not recipient count.

### 2. pgvector replaces Qdrant at current scale
The 7-day deduplication window holds at most ~3,500 active article vectors (500 articles/day × 7 days). pgvector's HNSW index handles this with <5ms query latency, eliminating Qdrant as a separate service. The schema includes a `cluster_updated_at` index to enable efficient 7-day window pruning. Qdrant remains the documented upgrade path at 50K+ vectors.

### 3. Deduplication uses concrete thresholds, not fuzzy heuristics
- Cosine similarity > **0.82** → cluster candidate
- Entity Jaccard overlap > **0.30** → confirms cluster membership
- Cosine > **0.88** vs delivered_items → candidate duplicate
- Fact delta score > **0** (any of 8 structured fields changed) → allow as follow-up
- Duplication penalty: **−30** (true dup) / **−5** (follow-up) applied to final score
These are starting thresholds requiring calibration in the first 2 weeks of operation. Suppressed items are logged (not just sent items) to enable threshold tuning.

### 4. Feedback URLs are HMAC-SHA256 signed
Email clients (especially iOS Mail) pre-fetch links — an unsigned feedback URL would trigger phantom votes. Every feedback link is signed with a server secret and a 48-hour TTL. The redirect service validates the signature before recording the event. Confirmation is a static page on OCI Object Storage (no server required).

### 5. Gmail 102KB hard clip is an operational constraint
Gmail silently truncates HTML emails above ~102KB. At 8-15 story cards per audience with full tracking links, this is a real risk. The design spec mandates: MJML output + CSS inlining + minification + a hard 100KB budget enforced at render time. Stories are dropped from the bottom (lowest-ranked) if the budget is exceeded, with a "See full briefing →" link to the OCI Object Storage archive.

## Top 3 Open Risks

### Risk 1: Source access and rate limits
Several high-value sources (The Information, SemiAnalysis, FT, Bloomberg) are behind hard paywalls with no API. Earnings transcripts require either a paid provider (Refinitiv, AlphaSense) or manual monitoring. **Mitigation:** Start with Bing News + GDELT + direct RSS for Tier 1-2 sources. Build a manual "source injection" endpoint for curators to paste in paywalled content during the first phase.

### Risk 2: Deduplication threshold calibration takes real data
The similarity thresholds (0.82 cosine, 0.30 Jaccard) are well-reasoned starting points but will produce false positives and false negatives in the first 2 weeks. Suppressing a real story looks like an editorial failure to executives. **Mitigation:** Start in "log-only" mode for the dedup filter for the first week — send everything, but log what would have been suppressed. Calibrate thresholds against that data before activating suppression.

### Risk 3: Executive email client diversity
The four recipients likely use a mix of iOS Mail (Apple Silicon MacBooks and iPhones), Gmail web, and possibly Outlook. Each handles CSS, dark mode, and image rendering differently. **Mitigation:** Use MJML + Litmus (or Email on Acid) to test rendering before launch. The design spec includes a full email client compatibility matrix and dark mode CSS overrides.

## Recommended Implementation Phases

### Phase 1 — MVP (Weeks 1–4)
**Goal:** One audience, real content, delivered daily. Validate the core loop.

- [ ] Ingestion: feedparser for top 10 RSS sources + Bing News API
- [ ] Normalization: entity extraction (spaCy NER), source metadata, timestamps
- [ ] Basic scoring: source credibility + timeliness only (no embeddings yet)
- [ ] Single audience: start with one executive (recommend Karan — clearest weights)
- [ ] LLM generation: Haiku for summaries, Opus for OCI Implications
- [ ] Email delivery: Postmark, raw HTML (no MJML yet), no tracking links
- [ ] Manual cron trigger (not scheduled yet)
- [ ] Validate content quality manually for 1 week before scheduling

**Exit criteria:** 5 consecutive days of on-time delivery, content approved by recipient

---

### Phase 2 — Full Team + Dedup (Weeks 5–8)
**Goal:** All 4 audiences, 7-day memory, no repetition, tracked links.

- [ ] pgvector + bge-small embeddings for all articles
- [ ] Deduplication pipeline (start in log-only mode, calibrate thresholds)
- [ ] All 4 audience profiles active with per-audience rendering
- [ ] 2-stage renderer (shared bundle → per-audience selection)
- [ ] Custom tracking redirect service + OCI Object Storage archive
- [ ] MJML email templates with per-persona design differentiation
- [ ] Daily cron scheduler + OCI Notifications for failure alerting
- [ ] HN Algolia + Reddit community signal ingestion

**Exit criteria:** Dedup accuracy >90% on test set, all 4 recipients receiving daily

---

### Phase 3 — Feedback Loop + Analytics (Weeks 9–12)
**Goal:** Close the loop. Make the briefing measurably better over time.

- [ ] Feedback widget in email (thumbs up/down, More/Less like this)
- [ ] HMAC-signed feedback URLs
- [ ] Analytics dashboard: CTR by section, top stories by audience, open rates
- [ ] Postmark open/click event ingestion into tracking_events table
- [ ] Manual source injection endpoint (for paywalled content)
- [ ] "Too repetitive" flag and suppression logging UI
- [ ] Subject line A/B test framework
- [ ] Newsletter ingest via Postmark Inbound (for email newsletters)

**Exit criteria:** At least 2 feedback signals/week from recipients, CTR baseline established

---

### Phase 4 — Intelligence Upgrades (Post-launch)
- Learned affinity model from click data (update audience profile weights automatically)
- Slack/Teams delivery format
- Self-service audience profile editor
- Expansion to additional OCI teams or Oracle business units
- Earnings transcript ingestion (via paid provider)
